用Obisidian编写文章

frontmatter

Quartz 还允许在笔记中指定称为frontmatter的附加元数据。

---
title: Example Title
draft: false
tags:
  - example-tag
---
 
The rest of your content lives here. You can use **Markdown** here :)

Quartz 原生支持的一些常见前置字段：

  • title：页面标题。如果没有提供，Quartz 将使用文件名称作为标题。
  • description：用于链接预览的页面描述。
  • permalink：页面的自定义 URL，即使文件路径发生变化也将保持不变。
  • aliases：此笔记的其他名称。这是一个字符串列表。
  • tags：此笔记的标签。
  • draft：是否发布页面。
  • date：表示笔记发布日期的字符串。使用YYYY-MM-DD格式。

  • priority: 用于查询日期信息的数据源。优先级最高。可能值为"frontmatter"、"git"和"filesystem"。默认为["frontmatter", "git", "filesystem"]。如果您依赖git日期，请确保defaultDateType设置modified为quartz.config.ts。
  • HardLineBreaks: 此插件会自动将 Markdown 文本中的单行换行符转换为 HTML 输出中的硬换行符。由于此插件不符合实际 Markdown 的语义，因此默认不启用。但如果您想与Obsidian保持一致，可以启用它。

部署到Github Pages

为了方便访问，我把生成的静态文件单独部署到kuhbz.github.io中，并且Giscus也使用这个仓库的discussions。

步骤一：创建 <你的用户名>.github.io 仓库

如果这个仓库还不存在，请先在你的 GitHub 上创建它。这是一个特殊的仓库，GitHub 会自动将它的 main (或 master) 分支部署为你的用户主页，网址就是 https://<你的用户名>.github.io。举例：我的用户名为kuhbz，那么仓库名就是kuhbz.github.io，访问地址就是https://kuhbz.github.io

步骤二：生成并配置部署密钥

为了让你当前仓库的 Action 能够安全地推送到 kuhbz.github.io 仓库，最安全的方法是使用部署密钥（Deploy Keys）。

  1. 生成 SSH 密钥对： 在你的本地电脑上打开终端，运行以下命令来生成一个新的 SSH 密钥对。当提示输入文件名时，可以输入 deploy_key，并且不要设置密码（直接按回车）。

    这会生成两个文件：deploy_key (私钥) 和 deploy_key.pub (公钥)。

  2. 添加公钥到 kuhbz.github.io 仓库：

    • 打开 deploy_key.pub 文件，复制里面的全部内容。
    • 进入 kuhbz.github.io 仓库的 GitHub 页面。
    • 点击 “Settings” → “Deploy keys” → “Add deploy key”。
    • 给它一个标题，比如 “Quartz Deploy”。
    • 将公钥内容粘贴到 “Key” 字段中。
    • 非常重要：勾选 “Allow write access” 复选框。
    • 点击 “Add key”。

  3. 添加私钥到你当前 Quartz 仓库：

    • 打开 deploy_key 文件（私钥），复制里面的全部内容。
    • 进入你存放 Quartz 源码的仓库的 GitHub 页面。
    • 点击 “Settings” → “Secrets and variables” → “Actions”。
    • 点击 “New repository secret”。
    • 将 Secret 的名字设置为 DEPLOY_KEY。
    • 将私钥内容粘贴到 “Secret” 字段中。
    • 点击 “Add secret”。

步骤三：创建 deploy.yml 工作流

现在，新建 .github/workflows/deploy.yml 文件。这个新版本的工作流使用了peaceiris/actions-gh-pages 这个流行的 Action 来处理跨仓库部署。

总结

完成以上步骤后：

  1. 保存你对 deploy.yml 文件的修改。
  2. 运行 npx quartz sync 来提交并推送你的更改。

这次推送会触发新的 GitHub Actions 工作流。它会：

  1. 构建你的 Quartz 网站，生成静态文件到 public 目录。
  2. 使用你配置的部署密钥，安全地将 public 目录下的所有内容推送到 kuhbz.github.io 仓库的 main 分支。

稍等片刻，网站就会成功部署在 https://kuhbz.github.io 上了。

quartz/cli/handlers.js 文件中的 handleSync 函数揭示了 npx quartz sync 命令的完整工作流程。

该命令本质上是围绕 git 命令的一个封装，旨在简化将本地内容与远程 git 仓库同步的过程。

以下是它的详细步骤：

  1. 备份内容 (stash)：

    • 命令首先会调用 stashContentFolder(contentFolder) 函数。这个函数的作用是使用 git stash push —include-untracked — path/to/content 命令，将 content 目录下的所有更改（包括未跟踪的文件）都暂存起来。
    • 这样做是为了保护用户的本地内容，避免在后续的 git pull 操作中产生冲突。

  2. 拉取更新 (pull)：

    • 如果用户没有使用 —no-pull 标志，命令会执行 gitPull(ORIGIN_NAME, QUARTZ_SOURCE_BRANCH)。
    • 这会运行 git pull —no-rebase —autostash origin v4 命令，从远程的 origin 拉取 v4 分支的最新更改。—autostash 选项会自动处理本地的非 content 目录的更改。

  3. 恢复内容 (pop)：

    • 拉取更新后，命令会调用 popContentFolder(contentFolder)，它会执行 git stash pop 来恢复之前暂存的 content 目录。
    • 如果 git pull 导致了冲突，git stash pop 可能会失败，这时 git 会提示用户手动解决冲突。

  4. 提交更改 (commit)：

    • 如果用户没有使用 —no-commit 标志，命令会自动创建一个 git 提交。
    • 它会执行 git add . 来暂存所有更改。
    • 然后，它会使用一个默认的提交信息（例如 “Quartz sync: Aug 17, 2025, 10:30 AM”）或者用户通过 -m 标志提供的自定义信息来执行 git commit。

  5. 推送更改 (push)：

    • 最后，如果用户没有使用 —no-push 标志，命令会执行 git push -uf ORIGIN_NAME ，将本地的提交强制推送到远程的 origin 仓库。

总结来说，npx quartz sync 通过自动化的 stash、pull、commit 和 push 操作，为用户提供了一个一站式的解决方案来同步他们的 Quartz 笔记。它简化了版本控制的复杂性，让不熟悉 git 的用户也能轻松地管理他们的数字花园。
